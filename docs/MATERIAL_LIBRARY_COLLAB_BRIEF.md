# 内置材料库协作说明

> 给 Claude Code / Codex 后续会话使用。
> 本文只整理用户需求和候选实现思路，不代表已经选定最终方案。

## 这件事的核心需求

用户想给当前 Shell Cup Game 增加一个**内置材料库**。

目标不是让老师每节课都提前手动导入 `.txt` 或图片文件夹，而是把常用课堂材料提前整理进程序里。老师上课时只需要在游戏里点几下，就能进入对应材料并开始游戏。

用户口中的“数据库”不要优先理解成复杂的数据系统。更贴近用户意思的说法是：

> 软件里提前放好一柜子课堂材料，老师点进去选大类、Level、Unit，然后直接开始玩。

## 用户期望的使用路径

用户当前明确表达的路径是：

```text
内置材料库
  闪卡
    Level
      Unit
        图片材料

  单词
    Level
      Unit
        文本单词材料
```

老师上课时的心智模型应该类似：

```text
Flashcards -> BS3 -> Unit 7 -> Start
```

或者：

```text
Words -> BS3 -> Unit 7 -> Start
```

这里的重点是**先分大类**：

- `Flashcards`：下面全部是用户整理好的图片材料，例如 `.png` / `.jpg`
- `Words`：下面全部是用户整理好的文本单词材料，例如 `.txt`

然后每个大类下面再按课程体系划分：

- Level，例如 `BS1`、`BS2`、`BS3`
- Unit，例如 `Unit 1`、`Unit 7`

## 单词和闪卡的关系

当前用户想要的是非常直观的两套材料入口：

- 闪卡 = 图片材料
- 单词 = 文本材料

不要默认假设一个单词必须绑定一张图片。
不要默认要求同一个 Unit 里同时存在文字和图片。
不要先把它设计成“同一套内容可切换文字模式 / 图片模式”的复杂结构。

用户想表达的是：

```text
点进 Flashcards，就看到所有已经整理好的图片闪卡材料。
点进 Words，就看到所有已经整理好的 txt 单词材料。
```

这两类材料可以彼此独立。即使名字上都叫 `BS3 Unit 7`，也不必在第一版里强制建立对应关系。

## 这件事不是什么

不要把这个需求一开始讲成复杂数据库设计。

复杂数据库通常会思考这些问题：

- 一个词条是否同时有文字、图片、音频、标签？
- 图片是否必须和单词一一对应？
- 同一套内容是否能切换成单词玩法和闪卡玩法？
- 缺图、缺文字时怎么办？
- 是否需要搜索、筛选、排序、编辑、导入导出？

这些问题以后可能有价值，但不是用户当前想先解决的核心。

当前核心是：

```text
老师不用临时准备材料。
老师能从软件里直接点到对应 Level / Unit。
老师能直接开始游戏。
```

## 可以建议的实现方式

下面只是候选方案，后续 Claude Code 不要直接替用户拍板。应该把方案讲清楚后，让用户决定。

### 方案 A：文件夹式内置材料库

用项目文件夹直接表达材料结构。

示意：

```text
data/library/
  flashcards/
    BS3/
      Unit 7/
        cat.png
        dog.png
        bird.png

  words/
    BS3/
      Unit 7.txt
```

优点：

- 最接近用户现在的理解方式
- 用户自己也容易维护材料
- 第一版实现最直接
- 不需要老师理解 JSON

缺点：

- 显示名称、排序、封面图等高级信息不够灵活
- Unit 名称主要依赖文件夹名 / 文件名

### 方案 B：文件夹 + 小说明文件

材料仍然放在文件夹里，但每个大类或每个 Level 可以有一个很小的说明文件，用来控制显示名称和排序。

示意：

```text
data/library/
  flashcards/
    BS3/
      manifest.json
      Unit 7/
        cat.png
        dog.png
```

优点：

- 仍然保留文件夹直觉
- 可以控制 Unit 顺序、显示名称、封面等
- 以后扩展更舒服

缺点：

- 比方案 A 多一点规则
- 用户或维护者需要知道说明文件的存在

### 方案 C：完整 JSON 材料索引

用 JSON 统一记录全部材料。

示意：

```text
BS3 Unit 7:
  type: flashcards
  files:
    - cat.png
    - dog.png
```

优点：

- 程序读取最稳定
- 排序、显示名、标签、封面、备注都能控制
- 未来如果做搜索或多教材体系会更强

缺点：

- 对当前需求可能过重
- 用户不容易直接维护
- 容易让问题变得像“数据库设计”，偏离用户最初想要的简单课堂入口

## 当前推荐倾向

可以向用户推荐从**方案 A：文件夹式内置材料库**开始。

理由：

- 它最接近用户说的“闪卡下面都是 png，单词下面都是 txt”
- 它最容易解释和检查
- 它和当前程序已有的能力天然接近：程序已经能读取 `.txt` 单词和扫描图片文件夹
- 它不阻止以后升级到方案 B

但这只是推荐，不是最终决定。后续应让用户拍板。

## 第一版可以先确认的问题

后续进入实现前，建议只问必要问题，不要一次抛太多技术细节。

优先确认：

1. Level 名称是否固定为 `BS1`、`BS2`、`BS3` 这类？
2. Unit 显示是否用 `Unit 1`、`Unit 2`、`Unit 7` 这种格式？
3. 闪卡图片是否只需要第一层文件夹，不递归子文件夹？
4. 单词 txt 是否每行一个词，或继续兼容 `1. apple` 这种编号格式？
5. 老师选完材料后，是直接开始游戏，还是先回到 Settings 页让老师调 `Answers`、`Cups`、`Rounds`、`Speed`？

## 实现时需要注意的项目约束

如果最终决定实现，需要遵守项目已有规则：

- UI 文字保持英文
- 资源路径必须兼容 PyInstaller
- 新增内置材料目录时，需要确认 `build.bat` 的 `--add-data` 是否包含该目录
- Windows 打包仍由用户在 Windows 端验证
- 不要引入新依赖
- 新 UI 仍需符合当前 cozy farming pixel UI + Champagne Farm 风格

## 给后续 AI 的协作提醒

这不是一个应该直接开写的功能。

先和用户确认：

- 用户是否接受“内置材料库”这个叫法
- 用户是否接受 `Flashcards / Words -> Level -> Unit` 这个入口
- 用户是否选择方案 A、B、C，或提出自己的调整

在用户拍板前，不要把某一种数据结构当作最终方案。
